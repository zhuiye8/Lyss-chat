## 架构概述

Lyss-chat 2.0 采用前后端分离的微服务架构，通过 API 网关统一管理服务访问，实现了系统的高内聚低耦合。系统主要分为前端应用、API 网关、核心服务、AI 引擎和数据存储五个部分。

### 设计原则

1. **关注点分离**：每个服务专注于特定的业务功能
2. **单一职责**：每个组件只负责一个功能领域
3. **接口隔离**：服务间通过定义良好的 API 进行通信
4. **可替换性**：组件可以独立升级或替换，不影响整体系统
5. **可扩展性**：系统可以通过添加新的服务实例水平扩展
6. **容错性**：单个服务故障不会导致整个系统崩溃

### 技术栈选型

1. **Web 框架**：gorilla/mux
   - 成熟稳定的 Go Web 框架
   - 灵活的路由匹配和中间件支持
   - 良好的性能和可维护性

2. **AI 引擎**：cloudwego/Eino
   - 专为 AI/LLM 应用开发设计的框架
   - 提供丰富的 AI 组件抽象和编排能力
   - 支持流式处理和工具调用

3. **数据库**：PostgreSQL
   - 强大的关系型数据库，支持复杂查询和事务
   - 良好的并发性能和可扩展性

4. **缓存**：Redis
   - 高性能的内存数据库，用于缓存和会话管理
   - 支持多种数据结构和持久化选项

5. **对象存储**：MinIO
   - 兼容 S3 API 的对象存储服务
   - 适用于存储文件和图片

## 组件详解

### 1. 客户端应用

基于 React 18 和 Ant Design X 构建的单页应用，主要包括以下组件：

- **用户界面**：页面布局、组件渲染和用户交互
- **状态管理**：使用 Zustand 管理应用状态
- **API 客户端**：基于 Axios 的 HTTP 客户端，负责与后端 API 通信
- **工具库**：通用工具函数和辅助组件

### 2. API 网关

基于 Go 语言实现的 API 网关，是系统的统一入口，主要功能包括：

- **认证授权**：验证用户身份和权限
- **请求路由**：将请求转发到相应的微服务
- **限流熔断**：防止系统过载和服务雪崩
- **日志监控**：记录请求日志和监控系统状态

### 3. 核心服务

系统的核心业务逻辑，每个服务都是独立的微服务，可以单独部署和扩展：

#### 3.1 用户服务

负责用户管理和认证授权，主要功能包括：

- 用户注册、登录和密码管理
- 基于 RBAC 的权限控制
- 多租户管理
- 用户配置和偏好设置

#### 3.2 模型服务

负责 AI 模型管理和 API 密钥管理，主要功能包括：

- 模型提供商管理
- 模型配置和参数管理
- API 密钥安全存储和管理
- 模型分发和访问控制

#### 3.3 聊天服务

负责对话管理和消息处理，主要功能包括：

- 对话创建和管理
- 消息处理和流式响应
- 多模态输入处理
- 上下文管理和历史记录

#### 3.4 代码服务

负责代码画布和代码执行，主要功能包括：

- 代码编辑器集成
- 语法高亮和自动完成
- 代码执行环境
- 代码分享和导出

### 4. Eino AI 引擎

基于 cloudwego/Eino 实现的 AI 引擎，提供 AI 组件抽象和编排能力：

- **组件抽象**：提供 ChatModel、Tool、ChatTemplate、Retriever 等 AI 应用常用组件
- **编排能力**：支持链式和图形编排，处理类型检查、流处理、并发管理等
- **流处理**：自动处理 AI 模型的流式响应，支持流的连接、合并、复制等操作
- **切面机制**：支持回调处理，用于日志、追踪、指标收集等横切关注点

### 5. 数据存储

系统的数据存储层，包括：

- **PostgreSQL**：主数据库，存储结构化数据
- **Redis**：缓存和会话管理
- **MinIO**：对象存储，用于存储文件和图片

### 用户认证流程

```
┌─────────┐      ┌─────────┐      ┌─────────┐      ┌─────────┐
│  客户端  │──1──▶│ API网关  │──2──▶│ 用户服务 │──3──▶│ 数据库   │
└─────────┘      └─────────┘      └─────────┘      └─────────┘
     ▲                                │                  │
     │                                │                  │
     └────────────5─────────────────4◀┘                  │
                                      │                  │
                                      └────────6─────────┘
```

1. 客户端发送登录请求
2. API 网关转发请求到用户服务
3. 用户服务查询数据库验证用户凭据
4. 数据库返回用户信息
5. 用户服务生成 JWT 令牌
6. 客户端接收令牌并存储

```
┌─────────┐      ┌─────────┐      ┌─────────┐      ┌─────────┐
│  客户端  │──1──▶│ API网关  │──2──▶│ 聊天服务 │──3──▶│ Eino AI │
└─────────┘      └─────────┘      └─────────┘      └─────────┘
     ▲                                │                  │
     │                                │                  │
     └────────────6─────────────────5◀┘                 4│
                                      │                  ▼
                                      │            ┌─────────┐
                                      └────7──────▶│ 数据库   │
                                                   └─────────┘
```

1. 客户端发送消息
2. API 网关转发请求到聊天服务
3. 聊天服务调用 Eino AI 引擎处理消息
4. Eino AI 引擎调用 AI 模型 API
5. Eino AI 引擎返回 AI 响应
6. 聊天服务将响应流式传输给客户端
7. 聊天服务将对话保存到数据库

## 通信机制

### 服务间通信

服务间通信采用以下机制：

1. **RESTful API**：基于 HTTP 的同步通信
2. **WebSocket**：用于实时通信，如聊天消息的流式传输

### 数据序列化

- **JSON**：用于 RESTful API 的请求和响应

## 扩展性设计

系统设计考虑了未来的扩展需求，主要体现在以下几个方面：

1. **服务水平扩展**：每个服务可以独立部署多个实例，通过负载均衡分发请求
2. **模块化设计**：系统按功能模块划分，可以方便地添加新模块
3. **插件化架构**：核心功能通过插件机制扩展，如添加新的模型提供商
4. **API 版本控制**：API 设计支持版本控制，确保向后兼容

## 高可用设计

为确保系统的高可用性，采取了以下措施：

1. **服务冗余**：关键服务部署多个实例
2. **负载均衡**：通过负载均衡器分发请求
3. **熔断机制**：防止服务雪崩
4. **限流机制**：防止系统过载
5. **数据备份**：定期备份数据，确保数据安全
6. **监控告警**：实时监控系统状态，及时发现并解决问题

## 性能优化

系统性能优化主要从以下几个方面考虑：

1. **缓存策略**：使用 Redis 缓存频繁访问的数据
2. **数据库优化**：合理设计索引，优化查询
3. **异步处理**：非关键操作采用异步处理
4. **资源池化**：使用连接池管理数据库连接
5. **压缩传输**：压缩 HTTP 响应，减少网络传输量
6. **CDN 加速**：静态资源通过 CDN 分发
